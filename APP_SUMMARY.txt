WHAT IS AIDEY?
================================================================================

AIDEY (Ang Gabay sa Dokumento at ID) is a Filipino government document and
ID guide — a journey-centered mobile companion for navigating Philippine
bureaucratic paperwork.

Tagline: "Because every opportunity begins with a document… and every
document should begin with AIDEY."

The app is NOT information-dump style. It walks users through each stage of
getting a government document, unlocking the next step only when ready.

Problems it solves:
- Scattered, outdated requirements across dozens of agencies
- "ID to get an ID" dependency chains
- Wasted trips to government offices with incomplete requirements
- Confusion around National ID and where to start

FEATURES
================================================================================

1. DOCUMENT & ID CATALOG
   - 33 Philippine government documents across 5 categories
   - Fuzzy search + speech-to-text dictation
   - Categories: Primary IDs, Social Security/Welfare, Professional/Tax/
     Employment, Clearances/Certifications, Civil Registry/Immigration

2. STEP-BY-STEP GUIDES
   - Prerequisites with cross-document dependency linking
   - Numbered process steps with estimated timelines
   - Progress saved locally and synced to cloud when signed in
   - Sections: Requirements, Process Steps, Upload

3. OWNERSHIP FLOW (per document)
   - WALA ("I don't have it") → AI help or step-by-step guide
   - MAYROON ("I have it") → capture/store ID photos

4. DOCUMENT UPLOAD & WALLET
   - Capture or pick photos of IDs
   - Local-first caching with Firebase Storage cloud sync
   - Multi-image per document, offline access

5. AI VIRTUAL ASSISTANT
   - Gemini-powered chat with structured JSON responses
   - Suggested reply chips, task checklists, step messages
   - Context-aware: knows guide progress, owned documents, locale
   - Can suggest opening maps or saving documents

6. OFFICE FINDER & DIRECTIONS
   - GPS-based nearest issuing office lookup
   - Turn-by-turn routing on interactive map
   - Google Routes API + Places API

7. BILINGUAL UI
   - Full English and Filipino support across app, guides, and AI

8. AUTH & CLOUD SYNC
   - Email/password sign-in gates cloud features
   - Syncs uploads, chat history, and guide progress across devices

  Localized guides in constants/document-guides/{en,fil}.ts.

DocumentOwnership
  Whether user has the document: mayroon (have) or wala (don't have).

User

Locale
  en-US | fil-PH. Affects UI, guides, AI responses, speech-to-text.

DirectionsRoute
  Map route from user location to office (polyline + distance/duration).

================================================================================
AI ASSISTANT (chat.ts)
================================================================================

Provider: Google Gemini via @google/genai
Model fallback chain:
  gemini-2.5-flash → gemini-2.5-flash-lite → gemini-3-flash → gemini-3.1-flash-lite

Structured JSON responses via AIDEY_RESPONSE_JSON_SCHEMA.

Context passed to AI:
  - documentLabel (current document being discussed)
  - ownedDocumentIds
  - documentGuideProgress
  - atOfficeProximity (within 25m of office)
  - checklist state
  - locale (en-US or fil-PH)

Behavior rules:
  - One step per reply
  - 2-5 suggestion chips per response
  - Map nearest office by default when relevant
  - Task checklists for multi-step guidance
  - save_document and open_maps action suggestions
  - Bilingual responses matching user locale

================================================================================
DATA SYNC PATTERN
================================================================================

All user data follows local-first + cloud sync:

1. Local storage: AsyncStorage with versioned keys per user (uid-scoped)
2. Cloud: Firebase Realtime Database for metadata
3. Files: Firebase Cloud Storage for images
4. Sync triggers: NetInfo reconnect, AppState active, auth state change
5. Merge strategy: compare updatedAt timestamps, push local-only to cloud
6. On auth user change: reset all stores and re-hydrate from cloud

Services using this pattern:
  - chat-sessions.ts (key: chat-sessions-v1:{uid})
  - document-uploads.ts (key: document-uploads-v2:{uid})
  - document-guide-progress.ts (key: document-guide-progress-v1:{uid})

HOW TO RUN / BUILD
================================================================================

Prerequisites: Node.js, pnpm, Expo CLI, EAS CLI, Android SDK, Firebase CLI

Development:
  pnpm install
  cp .env.example .env    # fill in API keys
  pnpm start              # expo start --dev-client
  pnpm start:go           # expo start (Expo Go)

Android:
  pnpm android            # scripts/run-android.sh
  pnpm android:build      # EAS development APK
  pnpm android:preview    # scripts/build-preview-apk.sh

Prebuild:
  pnpm prebuild           # expo prebuild --platform android
  pnpm prebuild:clean     # with --clean flag

Lint:
  pnpm lint               # expo lint

Firebase deploy:
  firebase deploy --only hosting:aidey --project glak-hackathon
  firebase deploy --only database,storage --project glak-hackathon

EAS build profiles (eas.json):
  development  — dev client APK, internal distribution
  preview      — internal APK without dev client
  production   — auto-increment version, production environment
